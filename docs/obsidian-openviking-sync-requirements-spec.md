# Obsidian 与 OV 记忆观察窗插件需求规格说明

## 1. 文档目的

本文定义 Obsidian 与 OpenViking（下文简称 `OV`）之间的记忆观察与受控回写插件的 v1 需求规格。目标是让实现者无需再做关键产品决策，可直接据此拆分任务并开始开发。

## 2. 产品定位

- `OV` 是唯一主库
- `Obsidian` 是投影视图、观察窗口和人工修正入口
- 插件仅支持 `Obsidian 桌面端`
- 插件通过 `OV 官方 HTTP API` 与本机或远端 OV 服务通信
- v1 不要求实时同步，采用 `启动同步 + 手动同步 + 60 秒轮询`

## 3. 范围与非目标

### 3.1 v1 范围

- 展示 OV memory 树中的目录摘要和叶子 memory 文件
- 默认投影根目录采用自动发现：
  - 当前 user space 下的 `viking://user/{space}/memories`
  - 当前 agent space 下的 `viking://agent/{space}/memories`
- 支持从 Obsidian 查看目录级 `L0/L1` 与文件级正文
- 仅允许编辑叶子 memory 文件正文
- 支持通过 correction memory 方式回写
- 支持远端删除观察、待确认删除和修订历史

### 3.2 非目标

- 不支持移动端
- 不支持真正对等双主同步
- 不支持直接覆盖 OV 的 `L0/L1`
- 不支持依赖未文档化的 memory 原地写接口
- 不支持附件同步

## 4. 核心对象

### 4.1 RemoteEntry

远端条目统一抽象为 `RemoteEntry`，可能是目录，也可能是文件。最小标识为：

- `uri`
- `isDir`
- `fs/stat.modTime`

说明：

- `fs/ls` 主要用于发现条目
- `fs/stat` 才是版本判断的权威来源

### 4.2 DirectorySummaryProjection

目录级投影，对应 OV 真实目录上的两个派生文件：

- `{dir}/.abstract.md`
- `{dir}/.overview.md`

在 Obsidian 中投影为可见的只读文件：

- `_dir.abstract.md`
- `_dir.overview.md`

### 4.3 MemoryFileProjection

文件级投影，对应 OV 中真实的 leaf memory 文件，例如：

- `mem_xxx.md`
- `profile.md`

只有这类投影允许编辑。

### 4.4 DraftCorrection

用户对可编辑 memory 文件的本地修改，尚未提交到 OV。

### 4.5 RevisionEvent

描述某条记忆生命周期中的结构化事件，例如远端更新、本地草稿、提交成功、提交失败、删除确认。

## 5. 文件投影模型

### 5.1 目录结构

默认投影目录：

```text
OpenViking/
  user/
    <space>/
      memories/
        _dir.abstract.md
        _dir.overview.md
        profile.md
        preferences/
          _dir.abstract.md
          _dir.overview.md
          mem_xxx.md
  agent/
    <space>/
      memories/
        _dir.abstract.md
        _dir.overview.md
        cases/
          _dir.abstract.md
          _dir.overview.md
          mem_xxx.md
  _deleted/
```

其中：

- `<space>` 为 OV 真实空间名，如 `default`、`ffb1327b18bf`
- `_dir.abstract.md` 对应远端 `{dir}/.abstract.md`
- `_dir.overview.md` 对应远端 `{dir}/.overview.md`
- 叶子文件保留远端原始文件名

### 5.2 层语义

- 目录级 `L0`：OV 自动生成摘要，只读
- 目录级 `L1`：OV 自动生成概览，只读
- 文件级正文：真实 memory 文件，唯一可编辑层

若用户修改 `L0/L1`，插件必须恢复远端内容并提示：

`L0/L1 are generated by OV and cannot be edited directly.`

## 6. Frontmatter 规格

### 6.1 通用字段

所有投影文件都包含以下字段：

```yaml
ov_uri: viking://user/default/memories/preferences/mem_xxx.md
ov_root: viking://user/default/memories
ov_scope: user
ov_space: default
ov_entry_type: memory_file
ov_layer: detail
ov_name: mem_xxx.md
ov_projection_version: 1
ov_remote_mod_time: 2026-03-20T10:00:00Z
ov_remote_size: 1024
ov_remote_exists: true
ov_synced_at: 2026-03-20T10:01:00Z
ov_last_seen_at: 2026-03-20T10:01:00Z
ov_status: synced
ov_deleted: false
ov_readonly: false
ov_managed_by: obsidian-openviking-plugin
```

### 6.2 L0/L1 专属字段

```yaml
ov_target_uri: viking://user/default/memories/preferences
ov_entry_type: directory_abstract | directory_overview
ov_editable: false
ov_generated_by: ov
```

### 6.3 memory 文件专属字段

```yaml
ov_entry_type: memory_file
ov_editable: true
ov_has_local_draft: false
ov_local_draft_updated_at:
ov_last_submit_at:
ov_last_submit_result:
ov_last_correction_uri:
ov_correction_target_uri: viking://user/default/memories/preferences/mem_xxx.md
```

## 7. 插件本地状态存储

插件使用自身持久化存储运行时真相，不把内部控制信息写入 frontmatter。

```ts
type PluginState = {
  version: 1
  settings: PluginSettings
  projections: Record<string, ProjectionState>
  revisions: Record<string, RevisionEvent[]>
}
```

说明：

- `projections` 以 `ovUri` 为 key
- `revisions` 每个 `ovUri` 保留最近 200 条事件
- 存储 `lastSyncedContent`、`selfWriteToken`、`lastRemoteHash`

补充约束：

- 目录级摘要和目录级概览分别有独立 `ProjectionState`
- 叶子 memory 文件也有独立 `ProjectionState`

frontmatter 和 state store 不一致时，以 `state store` 为准，并由插件修复 frontmatter。

## 8. 状态机

`ov_status` 固定取值：

- `synced`
- `stale_remote`
- `local_draft`
- `submit_pending`
- `submit_failed`
- `delete_pending`
- `deleted_remote`

状态规则：

- 只有 `memory_file` 可以进入 `local_draft`
- 有本地草稿时，远端更新不能覆盖本地 memory 文件正文
- 删除必须经过 `delete_pending`

## 9. 命令与交互

插件提供以下命令：

- `OpenViking: Sync now`
- `OpenViking: Submit correction`
- `OpenViking: Reset local draft`
- `OpenViking: Mark delete`
- `OpenViking: Confirm delete`
- `OpenViking: Show revision history`
- `OpenViking: Open linked correction`
- `OpenViking: Reveal in OpenViking path`

命令规则：

- `Submit correction` 仅对可编辑的 `memory_file` 可用
- `Reset local draft` 仅在 `local_draft` 或 `submit_failed` 时可用
- `Confirm delete` 仅在 `delete_pending` 时可用

## 10. 同步时序

### 10.1 启动

1. 读取 settings
2. 读取 state store
3. 执行 migration
4. 确保 `vaultFolder` 存在
5. 执行一次全量同步
6. 启动轮询定时器

### 10.2 轮询与手动同步

同步流程：

1. 遍历所有 `projectionRoots`
2. 调 `fs/ls(recursive=true)` 做条目发现
3. 对发现到的目录创建或刷新 `_dir.abstract.md` 和 `_dir.overview.md`
4. 对发现到的真实 Markdown 文件创建或刷新文件投影
5. 对每个候选条目调 `fs/stat` 获取规范化版本信息
6. 更新时：
   - 目录摘要和目录概览直接刷新
   - 若可编辑 memory 文件存在本地草稿，则只更新远端快照并记为 `stale_remote`
   - 否则刷新正文
7. 远端删除时移入 `_deleted/`

### 10.3 本地编辑监听

- 仅监听可编辑的 `memory_file`
- 当前正文与 `lastSyncedContent` 不同时，记为 `local_draft`
- 再次改回一致时，恢复 `synced`

## 11. Correction 提交流程

### 11.1 前提

- 当前文件必须是可编辑的 `memory_file`
- 必须存在 `ov_uri`
- 当前内容必须与 `lastSyncedContent` 不同

### 11.2 提交模板

提交给 OV 的结构化消息内容固定如下：

```text
Memory correction request

Target memory URI:
{{ov_uri}}

Original remote summary:
{{best_available_summary}}

Original remote content:
{{last_synced_file_content}}

Human-corrected content:
{{current_editor_content}}

Correction intent:
This is a human correction for the target memory. Keep the meaning aligned with the corrected content. Prefer producing one corrected long-term memory entry rather than multiple fragmented memories.

Source:
obsidian-openviking-plugin
```

其中 `best_available_summary` 的来源固定为：

1. 父目录 `overview`
2. 父目录 `abstract`
3. `lastSyncedContent` 的前若干字符摘要

原因是 leaf memory 文件本身不具备可直接读取的 `abstract/overview` 接口。

### 11.3 提交步骤

1. 状态置为 `submit_pending`
2. 创建临时 session
3. 提交 correction message
4. 调用 `extract`
5. 仅当结果中存在新的、与原 `ov_uri` 不同的 correction memory URI 时，才视为成功
6. 成功后建立 relation link
7. 刷新当前 URI 投影
8. 成功后恢复为 `synced`

失败时：

- 状态改为 `submit_failed`
- 保留草稿
- 记录失败原因和时间

## 12. Revision History 结构

```ts
type RevisionEvent = {
  id: string
  ovUri: string
  timestamp: string
  type:
    | "remote_created"
    | "remote_updated"
    | "remote_deleted"
    | "local_draft_created"
    | "local_draft_reset"
    | "submit_started"
    | "submit_succeeded"
    | "submit_failed"
    | "delete_marked"
    | "delete_confirmed"
  layer?: "l0" | "l1" | "detail"
  summary: string
  remoteModTime?: string
  correctionUri?: string
  diffSummary?: string
  errorMessage?: string
}
```

展示要求：

- 按时间倒序
- 默认展示最近 20 条
- 支持跳转到 `correctionUri`

## 13. 错误处理与恢复

### 13.1 OV 不可用

- 不修改现有投影
- 不清空 state
- 状态栏显示 `OV unavailable`

### 13.2 单条拉取失败

- 不阻断整轮同步
- 当前 URI 标记为 `stale_remote`
- 下轮继续重试

### 13.3 本地文件缺失

- 以 state store 为准
- 下轮同步自动重建

### 13.4 删除失败

- 保持 `delete_pending`
- 不移动文件
- 允许用户再次确认或取消

### 13.5 安全原则

- 不因同步失败删除 memory 文件草稿
- 不自动执行远端删除
- 不覆盖有本地草稿的 memory 文件正文
- 不允许直接写 L0/L1

## 14. 默认配置

```ts
type PluginSettings = {
  baseUrl: "http://localhost:1933"
  apiKey?: string
  autoDiscoverRoots: true
  projectionRoots: []
  vaultFolder: "OpenViking"
  pollIntervalSec: 60
}
```

说明：

- `projectionRoots` 为空时，插件启动时自动发现当前 user/agent space 对应的 `memories` 根目录
- 例如当前实例会解析为 `viking://user/default/memories` 与 `viking://agent/ffb1327b18bf/memories`

## 15. MVP 验收标准

- OV 新增记忆后，Obsidian 能在下一轮同步内看到
- OV 更新记忆后，Obsidian 能刷新当前投影并追加历史事件
- 用户只能编辑真实 memory 文件，不能编辑目录摘要与概览
- 用户可将 memory 文件草稿提交为 correction memory
- correction 成功后能记录 `correctionUri` 并建立 relation link
- 远端删除不会直接让本地痕迹消失，而会进入 `_deleted/`
- 插件重启后，state store、frontmatter、历史视图保持一致
